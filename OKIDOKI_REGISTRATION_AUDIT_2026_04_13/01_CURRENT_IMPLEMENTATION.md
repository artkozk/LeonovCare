# 1. Текущее состояние реализации

## Что уже есть в боте

1. В профиле ученика есть выбор режима:
- `Автозаполнить по ссылке на договор`;
- `Заполнить вручную`.
- Код: `bot/app/handlers/user.py` (ветка `cb_profile_start_mode`).

2. В авто-режиме первый шаг — `contract_lookup`:
- бот ждёт ссылку;
- проверяет только формат `http/https`;
- дальше вызывает prefill по ссылке.
- Код: `bot/app/handlers/user.py` (`st_profile`, блок `if step == "contract_lookup"`).

3. Prefill по договору уже многоуровневый:
- поиск в `enrollment_contracts` по `sign_url/contract_id`;
- fallback на последнюю запись договора пользователя;
- fallback на live lookup в OkiDoki API по `contract_id`.
- Код: `bot/app/handlers/user.py` (`_prefill_profile_from_contract_url`, `_prefill_profile_from_contract_api`).

4. После prefill бот дозапрашивает только недостающие поля:
- `ФИО`, `username`, `направление`, `этап`, `тариф`, `предоплата/постоплата`, `ссылка`.
- Код: `bot/app/handlers/user.py` (`_profile_next_step`, `_profile_continue_wizard`).

5. Если по ссылке ничего не найдено:
- бот пишет, что данные не найдены в OkiDoki;
- сохраняет ссылку;
- продолжает ручной wizard.

## Где сейчас не хватает вашего целевого поведения

1. Нет отдельной строгой проверки «наш договор / не наш договор» по системным полям с явным verdict.

2. Сейчас fallback в ручной режим происходит при пустом prefill, но без чёткой классификации причины:
- невалидная ссылка;
- договор не наш;
- OkiDoki недоступен;
- неполные данные.

3. Нет отдельного read-only режима для OkiDoki именно для этого процесса:
- в коде есть и чтение, и создание договора (`create_contract`) в другом потоке.

4. Поля «сколько заплатил / сколько заплатит» заполняются из текущих эвристик prefill, но не как формальный обязательный контрактный блок с валидацией качества.
