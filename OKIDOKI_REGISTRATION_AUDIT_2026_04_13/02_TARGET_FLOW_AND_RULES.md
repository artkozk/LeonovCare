# 2. Целевой flow и правила валидации

## Целевой пользовательский путь

1. Пользователь нажимает `Я ученик`.
2. Бот просит ссылку на договор.
3. Бот валидирует ссылку:
- формат;
- извлечение `contract_id`.
4. Бот делает только read-only запросы в OkiDoki по договору.
5. Бот определяет, что договор «наш» по системным полям.
6. Если договор «наш»:
- автоматически подставляет `ФИО`, `Telegram`, `тариф`, `дата обучения`, `сколько заплатил`, `сколько заплатит`.
7. Если часть полей отсутствует:
- бот спрашивает только недостающее.
8. Если договор невалидный или не наш:
- бот пишет `Пройди ручную регистрацию`;
- отправляет заявку на модерацию ментору.

## Правило по username (новое)

1. Username вручную не спрашивается.
2. Username должен приходить из полей договора OkiDoki (`Телеграм клиента`).
3. Если username в договоре отсутствует:
- считаем договор неподходящим для авто-регистрации;
- отправляем в ручную модерацию.
4. Проверка соответствия:
- username из договора должен совпадать с username пользователя в Telegram.
5. Исключение для теста:
- если аккаунт пользователя `@artkozk`, mismatch допускается для проверки парсинга чужих договоров.

## Правила «наш договор»

1. Есть совпадение по разрешённому шаблону/системному идентификатору.
2. Присутствует минимальный обязательный набор системных полей.
3. Значения полей проходят базовые sanity-check:
- валидная дата;
- корректный тип тарифа;
- числовые поля оплаты в допустимых диапазонах.

## Read-only ограничения по OkiDoki

1. Для сценария регистрации по ссылке запрещаем:
- create;
- update;
- delete.

2. Разрешаем только:
- чтение карточки договора;
- чтение PDF (при необходимости extraction hints).

3. Любая попытка write-операции в этом флоу должна:
- блокироваться;
- логироваться как security событие.

## Fallback-поведение

1. Невалидная ссылка:
- сообщение пользователю + ручная регистрация.

2. Ссылка валидная, но договор не наш:
- сообщение пользователю + ручная регистрация + модерация.

3. OkiDoki временно недоступен:
- не блокировать пользователя;
- ручная регистрация + причина в логах и в заявке.

## Ограничение текущего состояния без правок кода

1. Этот документ фиксирует целевые правила.
2. Автоматическое enforcement новых username-правил и полного отказа от ручного username-ввода потребует кода.
3. По вашему требованию сейчас код не изменяется, поэтому это зафиксировано как контракт на следующую итерацию.
